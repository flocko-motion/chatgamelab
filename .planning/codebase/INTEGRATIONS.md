# External Integrations

**Analysis Date:** 2026-03-08

## APIs & External Services

**AI Platforms (pluggable, user-supplied API keys):**

- **OpenAI** - Text generation, image generation, TTS, STT
  - SDK/Client: Custom HTTP client via `server/apiclient/` (no official SDK)
  - Base URL: `https://api.openai.com/v1`
  - Endpoints used: `/responses` (Responses API), `/images/generations`, `/audio/speech`, `/audio/transcriptions`, `/models`
  - Models: gpt-5.2, gpt-5.1, gpt-5-mini (text); gpt-image-1.5 (image); gpt-4o-mini-tts (TTS); gpt-4o-mini-transcribe (STT)
  - Auth: API key stored per-user in DB (`api_keys` table) or system-level free-use key
  - Implementation: `server/game/ai/openai/`

- **Mistral AI** - Text generation, image, STT
  - SDK/Client: Custom HTTP client via `server/apiclient/`
  - Base URL: `https://api.mistral.ai/v1`
  - Endpoints used: `/conversations`, `/models`, `/files`, `/audio/transcriptions`
  - Models: mistral-large-latest, mistral-medium-latest, mistral-small-latest; voxtral-mini-latest (STT)
  - Auth: API key stored per-user in DB
  - Implementation: `server/game/ai/mistral/`

- **Mock AI** - Development/testing only
  - No external calls; returns static responses
  - Implementation: `server/game/ai/mock/`

**AI Platform Abstraction:**
- All platforms implement the `AiPlatform` interface defined in `server/game/ai/ai.go`
- Platform capabilities: `ExecuteAction`, `ExpandStory`, `GenerateImage`, `GenerateAudio`, `Translate`, `ListModels`, `GenerateTheme`, `TranscribeAudio`, `ToolQuery`
- Platform selection: `openai`, `mistral`, `mock` (constants in `server/game/ai/ai.go`)

## Data Storage

**Databases:**
- PostgreSQL (version 18 in production, 18 in Docker)
  - Connection env vars: `DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_DATABASE`, `PORT_POSTGRES`
  - Client: `github.com/lib/pq` (driver) + `sqlc`-generated query layer
  - Query layer: `server/db/sqlc/` (generated from `server/db/queries/` + `server/db/schema.sql`)
  - Migrations: custom in-house system, sequential numbered SQL files in `server/db/migrations/`, version tracked in `system_settings.schema_version`
  - Schema auto-initialized on first run (empty DB detection)

**File Storage:**
- Local in-memory image cache only (`server/game/imagecache/cache.go`) — images are persisted to DB as bytes, not to a file system or object store
- No external file storage service detected

**Caching:**
- In-process image cache in `server/game/imagecache/` with 5-minute TTL and 30-second cleanup interval
- No Redis or external cache

## Authentication & Identity

**Auth0 (production):**
- Provider: Auth0 (external SaaS identity provider)
- Backend validation: `github.com/auth0/go-jwt-middleware/v2` with RS256 JWKS validation
  - JWKS URL derived from `AUTH0_DOMAIN` env var
  - Audience validated against `AUTH0_AUDIENCE` env var
  - Implementation: `server/api/httpx/auth.go` (`Authenticate` middleware)
- Frontend SDK: `@auth0/auth0-react` v2.11.0 (configured in `web/src/providers/AppProviders.tsx`)
  - Config: `web/src/config/auth0.ts`
  - Auth0 tokens stored in `localStorage` (`cacheLocation: "localstorage"`) with refresh token rotation
- Auth flow: Auth0 issues JWT → backend validates → user looked up by `auth0_id` in DB → user object attached to request context

**Dev JWT (development only):**
- HS256 signed JWTs generated by the server itself
- Signing secret: `DEV_JWT_SECRET` env var
- Implementation: `server/api/auth/jwt.go`
- Used when `AUTH0_DOMAIN` is not configured

**Participant Tokens (workshop participants):**
- Simple token-based auth for workshop participants (no Auth0 required)
- Tokens prefixed `participant-`, stored in DB and validated via `db.GetUserByParticipantToken`
- Token delivered via session cookie (`cgl_session`) or Bearer header
- Implementation: `server/api/httpx/auth.go`

**Auth middleware priority (checked in order):**
1. Participant token (`participant-` prefix)
2. CGL dev JWT (HS256, signed with `DEV_JWT_SECRET`)
3. Auth0 JWT (RS256, validated via JWKS)

**Token sources accepted by middleware:**
- `Authorization: Bearer <token>` header
- `cgl_session` cookie (participant and dev sessions)
- `?token=` query param (for SSE EventSource connections that can't set headers)

## Monitoring & Observability

**Error Tracking:**
- Sentry / GlitchTip (Sentry-compatible, optional)
  - Backend: `github.com/getsentry/sentry-go` v0.42.0, initialized in `server/telemetry/telemetry.go`
    - Enabled when `SENTRY_DSN_BACKEND` env var is set
    - Sample rate: 100% errors, 10% traces
  - Frontend: `@sentry/react` v10.38.0, initialized in `web/src/config/sentry.ts`
    - Enabled when `SENTRY_DSN_FRONTEND` / `VITE_SENTRY_DSN_FRONTEND` is set
    - Sample rate: 100% errors, 10% traces

**Logs:**
- Backend: structured logging via custom `server/log/` package (wraps stdlib `slog`), with optional Sentry integration for error-level messages
- Frontend: `web/src/config/logger.ts` + custom Vite plugin (`web/plugins/browser-logs`) that forwards `console.*` calls to the dev server terminal

## CI/CD & Deployment

**Hosting:**
- Docker Compose-based self-hosted deployment
- Production images published to GitHub Container Registry (`ghcr.io/flocko-motion/chatgamelab-*`)
- Services: `chatgamelab-db` (PostgreSQL), `chatgamelab-backend` (Go binary), `chatgamelab-web` (nginx static)

**Reverse Proxy:**
- nginx 1.27-alpine serves the frontend static files and proxies `/api/` to the backend
- nginx config template: `web/docker/nginx.conf.template`; processed at container start by `web/docker/entrypoint.sh`

**CI Pipeline:**
- GitHub Container Registry used for image storage, suggesting GitHub Actions (CI config not explored)

**Build process:**
- Backend: `go build` with `-ldflags` for version injection (`GitCommit`, `Version`, `BuildTime`)
- Frontend: `tsc -b && vite build`; API types auto-generated via `npm run gen:api` (calls `swagger-typescript-api`) before each production build
- Build scripts: `build.sh`, `run-dev.sh`, `run-prod.sh`, `run-dev-client.sh` at repo root

## Webhooks & Callbacks

**Incoming:**
- Auth0 redirect callbacks:
  - `GET /auth/login/auth0/callback` (login)
  - `GET /auth/logout/auth0/callback` (logout)
  - Handled by `@auth0/auth0-react` on the frontend (SPA redirect)

**Outgoing:**
- AI platform API calls (OpenAI, Mistral) initiated from backend game session logic
- No webhook delivery to external systems detected

## Environment Configuration

**Required env vars (all services):**
```
# Database (backend)
DB_HOST, DB_USER, DB_PASSWORD, DB_DATABASE, PORT_POSTGRES

# Backend
PORT_BACKEND
AUTH0_DOMAIN, AUTH0_AUDIENCE

# Frontend (via window.__APP_CONFIG__ in prod / VITE_* in dev)
API_BASE_URL, AUTH0_DOMAIN, AUTH0_CLIENT_ID, AUTH0_AUDIENCE
```

**Optional env vars:**
```
DEV_JWT_SECRET          # Enables dev JWT auth; backend only
ADMIN_EMAILS            # Comma-separated emails auto-promoted to admin
SENTRY_DSN_BACKEND      # Backend error reporting
SENTRY_DSN_FRONTEND     # Frontend error reporting (prod)
VITE_SENTRY_DSN_FRONTEND # Frontend error reporting (dev)
BACKUP_ENABLED, BACKUP_SSH_HOST, BACKUP_SSH_PORT, BACKUP_SSH_USER, BACKUP_SSH_KEY_PATH, BACKUP_PATH  # DB backup
```

**Secrets location:**
- Development: `.env` file at repo root (gitignored)
- Production: set via systemd `Environment=`, Docker `--env-file`, or hosting provider secrets; SSH backup key via Docker secrets (`/run/secrets/backup_key`)
- CLI tool config: `~/.chatgamelab/config.yaml` (stores server URL, JWT, and per-platform AI API keys)
